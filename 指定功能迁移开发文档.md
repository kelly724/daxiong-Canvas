# 指定功能迁移开发文档

> 目标：把 AI-CanvasPro 中适合迁移的节点能力，按当前 `daxiong-Canvas` 的架构落地为可开发、可验收、可分阶段上线的方案。本文只写开发方案，不直接修改业务代码。

## 1. 本次迁移范围

本次迁移包含 7 类能力：

| 优先级 | 目标能力 | 目标节点 / 通道 | 说明 |
| --- | --- | --- | --- |
| P1 | 文本生成 | `ai-text` | 生成文案、脚本、标题、分镜文字、提示词草稿 |
| P1 | 源文本 | `source-text` | 短期复用现有 `prompt` / `smart-prompt` 能力；长期可升级为一等“源文本”节点 |
| P1 | Codex CLI 生图 | `codex_cli` Provider | 当前没有一等 Codex 节点，迁移为本机生图 Provider |
| P1 | 分镜脚本 | `storyboard-script` | 生成镜头表、画面提示词、视频提示词、节奏规划 |
| P2 | 故事板 / 分镜板 | `storyboard` | 管理分镜格、镜头格、图片格，服务短视频生产链路 |
| P2 | 评论 / 便签 | `comment-note` | 画布批注、说明、协作备注 |
| P2 | 场景检测 | `scene-detection` | 从视频或素材中识别镜头 / 场景，生成裁剪段或分镜输入 |

## 2. 当前项目架构约束

`daxiong-Canvas` 当前不是前后端分离的大型工程，而是：

- 后端：`main.py` 单文件 FastAPI 应用。
- 前端：`static/js/canvas.js` 负责主画布，`static/js/smart-canvas.js` 负责智能画布。
- 数据：画布节点和连线以 JSON 方式保存，接口集中在 `/api/canvases`。
- 已有生成通道：
  - 图片任务：`POST /api/canvas-image-tasks`，`GET /api/canvas-image-tasks/{task_id}`。
  - 视频生成：`POST /api/canvas-video`。
  - LLM 文本 / 多模态：`POST /api/canvas-llm`。
  - Provider 配置：`/api/providers`、`/api/providers/test-connection`、`/api/providers/fetch-models`。

所以迁移原则是：

1. 不照搬 AI-CanvasPro 的模块化目录。
2. 前端第一阶段优先接入 `static/js/canvas.js` 主画布。
3. 智能画布 `static/js/smart-canvas.js` 做最小兼容，不强行一次性完整重写。
4. 后端接口继续挂在 `main.py`，但 Codex CLI 这种本机运行逻辑建议拆到 `services/codex_cli_service.py`，避免继续膨胀 `main.py`。
5. 节点数据必须能被现有画布保存 / 读取，不破坏老画布。

## 3. 总体迁移设计

### 3.1 节点类型分层

本次新增节点按用途分 4 层：

| 层级 | 节点 / 通道 | 作用 |
| --- | --- | --- |
| 输入层 | `source-text`、现有图片 / 视频 / prompt 节点 | 提供文字、图片、视频素材 |
| 生成层 | `ai-text`、现有 generator / video / LLM 节点、Codex CLI Provider | 生成文本、图片、视频 |
| 编排层 | `storyboard-script`、`scene-detection` | 把素材转成镜头表、场景段、提示词 |
| 呈现层 | `storyboard`、`comment-note` | 展示分镜格、批注、说明和项目管理信息 |

### 3.2 推荐链路

短视频生产链路：

```text
source-text / prompt
  -> ai-text
  -> storyboard-script
  -> storyboard
  -> generator / Codex CLI Provider
  -> video
```

视频素材反推链路：

```text
source-video
  -> scene-detection
  -> storyboard-script
  -> storyboard
  -> ai-text / generator / video
```

文案到图片链路：

```text
source-text
  -> ai-text
  -> generator(provider=codex_cli 或其他图片 Provider)
```

批注链路：

```text
comment-note
  -> 默认不参与生成
  -> 用户勾选 includeAsContext 后，可进入 ai-text / storyboard-script 上下文
```

### 3.3 节点保存兼容

现有画布保存本质是保存 `nodes` 和 `connections` JSON。新增节点不要引入独立数据库表，第一阶段直接存在画布 JSON 中。

节点基础字段统一如下：

```json
{
  "id": "node_xxx",
  "type": "source-text",
  "x": 100,
  "y": 100,
  "w": 320,
  "h": 220,
  "title": "源文本",
  "createdAt": "2026-06-09T00:00:00Z",
  "updatedAt": "2026-06-09T00:00:00Z"
}
```

老画布兼容规则：

- 老节点不加字段，照常读取。
- 新节点如果前端暂时不认识，至少渲染成“未知节点卡片”，不能导致整个画布空白。
- 新字段使用可选字段，后端保存接口不做强校验。

## 4. 功能一：`source-text` 源文本节点

### 4.1 迁移目标

`source-text` 是“素材层文本”，不是“生成指令节点”。它适合放：

- 原始文案。
- 剧本段落。
- 人物设定。
- 参考提示词。
- 旁白文本。
- 甲方需求。

短期实现：主画布新增 `source-text` 节点；智能画布先复用 `smart-prompt`，用字段区分来源。

长期实现：智能画布也增加一等 `source-text` 节点。

### 4.2 前端改造

目标文件：

- `static/js/canvas.js`
- `static/js/smart-canvas.js`
- 如有样式拆分，可补充 `static/css` 中对应画布样式文件。

主画布新增函数：

```js
function addSourceTextNode(x, y, options = {}) {}
function renderSourceTextNode(node) {}
function updateSourceTextNode(id, patch) {}
function getTextFromSourceNode(node) {}
```

节点 UI：

- 标题栏：默认“源文本”。
- 正文：`textarea` 或 `contenteditable`。
- 底部信息：字数统计、用途标签。
- 输出口：允许连接到 `ai-text`、`storyboard-script`、图片生成、视频生成、LLM 节点。

节点字段：

```json
{
  "type": "source-text",
  "title": "源文本",
  "content": "",
  "contentHtml": "",
  "role": "reference",
  "tags": [],
  "charCount": 0,
  "includeAsContext": true
}
```

`role` 可选值：

| 值 | 说明 |
| --- | --- |
| `reference` | 参考资料 |
| `script` | 剧本 |
| `prompt` | 提示词 |
| `brief` | 需求说明 |
| `caption` | 旁白 / 字幕 |

### 4.3 后端改造

第一阶段不需要新增专用 API。原因：

- `source-text` 不执行生成任务。
- 当前 `/api/canvases` 已能保存节点 JSON。

只需要确保：

- 保存画布时不要过滤未知 `type`。
- 读取画布时保留 `source-text` 所有字段。

### 4.4 链路规则

允许输出到：

- `ai-text`
- `storyboard-script`
- `generator`
- `video`
- `llm`
- `rh`
- `comfy`

上下文拼接格式建议：

```text
[源文本: 节点标题]
节点内容...
```

## 9. 功能六：`scene-detection` 场景检测节点

### 9.1 迁移目标

`scene-detection` 用于从视频中识别场景或镜头段落。

它不是视频生成节点，而是视频理解 / 拆分节点。

### 9.2 两阶段实现

第一阶段：轻量语义场景检测。

- 复用当前 `/api/canvas-llm` 里已有的视频抽帧能力。
- 按固定间隔抽帧。
- 让 LLM 根据帧内容归纳场景段落。
- 优点：不需要安装 OpenCV。
- 缺点：不是严格意义的视频剪辑点检测。

第二阶段：真实镜头切点检测。

- 引入 `ffmpeg` + `opencv-python`。
- 用直方图差异 / 边缘差异检测镜头切换。
- 再把每段关键帧送给 LLM 总结。
- 优点：更适合后期工作流。
- 缺点：依赖更重，Mac 新手安装成本更高。

建议先做第一阶段。

### 9.3 数据结构

```json
{
  "type": "scene-detection",
  "title": "场景检测",
  "videoNodeId": "",
  "videoUrl": "",
  "provider": "llm-frame-sampling",

  "sampleInterval": 2,
  "running": false,

  "lastError": ""
}
```

## 10. 功能七：Codex CLI 生图 Provider

### 10.1 迁移目标

当前项目没有一等 Codex 节点，也没有 Codex CLI 本机生图通道。

迁移目标不是新增一个复杂节点，而是先把 Codex CLI 做成图片生成 Provider：

```text
generator 节点
  -> provider = codex_cli
  -> model = codex-cli/gpt-image-2
  -> 后端调用本机 codex exec
  -> 返回图片结果
```

这样能最大化复用当前 `generator` 节点和 `/api/canvas-image-tasks` 的思路。

### 10.2 后端新增文件

建议新增：

```text

  codex_cli_service.py
```

如果暂时不想建 `services` 目录，也可以放在根目录：


但推荐 `services/`，因为 Codex CLI 涉及：

- 命令检测。
- 登录检测。
- 子进程运行。
- 任务目录。
- 取消任务。
- 输出文件扫描。

这些不适合继续堆在 `main.py`。

### 10.9 generator 节点接入

当前主画布已有 generator 节点和 `/api/canvas-image-tasks`。迁移时推荐：

1. generator 节点新增 Provider 选项 `codex_cli`。
2. 当 `provider !== "codex_cli"`，继续走原 `/api/canvas-image-tasks`。
3. 当 `provider === "codex_cli"`，走 `/api/codex-cli/imagegen/start`。
4. 轮询 `/api/codex-cli/imagegen/{task_id}`。
5. 成功后把图片 URL 写回 generator 节点输出。

不要第一阶段新建 `codex-image` 节点，避免节点体系变复杂。

## 17. 风险与取舍

### 17.1 AI-CanvasPro 代码不能直接复制

AI-CanvasPro 中部分节点源码是混淆 / 压缩后的代码，不适合直接复制到当前项目。

正确方式：

- 学习产品结构和字段。
- 复用节点概念。
- 按 `daxiong-Canvas` 现有风格重新实现。

### 17.2 Codex CLI 依赖用户本机环境

Codex CLI Provider 是否可用，取决于：

- Mac 是否安装 Codex CLI。
- 是否登录。
- 当前版本是否支持 `codex exec`。
- 是否支持图片输入 / 输出。
- 当前项目目录是否可写。

所以状态检测和错误提示必须先做，不能直接在前端给一个“生成”按钮就结束。

### 17.3 场景检测不要一开始上重依赖

OpenCV 方案更准确，但对新手 Mac 安装环境更麻烦。

建议：

1. 第一阶段：视频抽帧 + LLM 语义分析。
2. 第二阶段：再引入 OpenCV / ffmpeg 的真实切点检测。

### 17.4 智能画布不要一次性全量迁移

当前主画布和智能画布是两套 JS 逻辑。

建议：

- 主画布先完整支持新增节点。
- 智能画布先复用 `smart-prompt` 接住 `source-text`。
- 等主画布稳定后，再把 `ai-text`、`storyboard-script` 迁到智能画布。

## 19. 最小可用版本定义

如果只做最小可用版本，必须包含：

- `source-text`
- `ai-text`
- `storyboard-script`
- Codex CLI Provider

原因：

- `source-text` 解决原始内容输入。
- `ai-text` 解决文案 / 脚本生成。
- `storyboard-script` 解决短视频生产链路的核心结构化输出。
- Codex CLI Provider 解决本机生图能力。

可以暂缓：

- `storyboard`
- `comment-note`
- `scene-detection`

但不建议长期缺失，因为这三项会明显提升画布的可用性和生产链路完整度。

## 21. 结论

这批功能适合迁移，但不能按 AI-CanvasPro 的代码结构直接搬。当前项目的正确路线是：

1. 用主画布 `canvas.js` 先承接新增节点。
2. 用现有 `/api/canvas-llm` 承接 `ai-text`。
3. 用新增 `/api/storyboard-script/generate` 承接结构化分镜。
4. 把 Codex CLI 作为 generator 的本机图片 Provider。
5. 场景检测先做轻量版，再升级真实切点检测。

这样改动最小，链路完整，也符合当前 `daxiong-Canvas` 的 FastAPI + 静态前端框架。
