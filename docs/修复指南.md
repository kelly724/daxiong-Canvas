# 修复指南

## RunningHub 高清视频返回 Invalid URL

### 现象

RunningHub 视频节点报错：

```text
RunningHub 视频生成成功但没有返回视频：
{'taskId': '', 'status': '', 'errorCode': '1001', 'errorMessage': 'Invalid URL, please check your link | 请求链接无效，请检查您的调用链接', ...}
```

这个错误表面上像是“生成成功但没有视频输出”，实际不是下载视频失败，而是请求走错了 RunningHub 接口。

### 根因

`runninghub/2047787809091620866` 不是标准 OpenAPI 视频模型 endpoint，不能拼成：

```text
/openapi/v2/runninghub/2047787809091620866
```

这个 ID 是 RunningHub 的高清修复 AI App。前端已经按业务语义把它当“视频高清”模型展示：

- `basic`：走工作流 `2019292222763573249`
- `quality`：走 AI App `2047787809091620866`
- `sharp`：走 AI App `2047787809091620866`

后端之前没有给这个模型做专门分流，而是把它当普通 OpenAPI 模型处理，所以 RunningHub 返回 `errorCode=1001 Invalid URL`。

另外，视频节点前端会把视频参考放在 `payload.videos`，而不是 `payload.images`。后端如果只看 `images`，也会漏掉真正的视频输入。

### 已修复

修复位置：`main.py`

新增专门处理逻辑：

- 常量：
  - `RUNNINGHUB_HD_VIDEO_MODEL = "runninghub/2047787809091620866"`
  - `RUNNINGHUB_HD_VIDEO_APP_ID = "2047787809091620866"`
  - `RUNNINGHUB_HD_VIDEO_BASIC_WORKFLOW_ID = "2019292222763573249"`
- `generate_runninghub_video()` 先判断是否为高清模型。
- 高清模型走 `generate_runninghub_hd_video()`。
- 输入视频优先从 `payload.videos` 读取。
- 本地视频先通过 `/task/openapi/upload` 上传，拿到 RunningHub 的 `fileName`。
- `basic` 提交到 `/task/openapi/create`，传 `workflowId=2019292222763573249`。
- `quality/sharp` 提交到 `/task/openapi/ai-app/run`，传 `webappId=2047787809091620866`。
- 查询结果统一走 `/task/openapi/outputs`。

### 防复发规则

以后接 RunningHub 模型时，先判断模型 ID 属于哪一类：

| 类型 | 示例 | 正确接口 |
|---|---|---|
| 标准 OpenAPI endpoint | `rhart-video/video-upscaler` | `/openapi/v2/...` |
| RunningHub AI App ID | `2047787809091620866` | `/task/openapi/ai-app/run` |
| RunningHub 工作流 ID | `2019292222763573249` | `/task/openapi/create` |

不要仅凭 `video_models` 里的字符串就直接拼 `/openapi/v2/{model}`。如果模型值里是数字 ID，或者形如 `runninghub/<数字ID>`，必须先确认它是 App、Workflow，还是标准 OpenAPI endpoint。

### 调试检查清单

遇到 RunningHub `Invalid URL` 时，按下面顺序查：

1. 查报错里的 `errorCode` 和 `errorMessage`。
2. 查当前模型 ID 是否是普通 OpenAPI endpoint。
3. 如果是 App 或 Workflow，确认是否走了 `/task/openapi/...`。
4. 查输入媒体是在 `payload.images` 还是 `payload.videos`。
5. 对本地素材，确认是否先上传到 RunningHub 并取得 `fileName`。
6. 查询任务输出时使用 `/task/openapi/outputs`，不要混用 `/openapi/v2/query`。

### 验证方式

不消耗 RunningHub 点数的本地验证：

```powershell
python\python.exe -m py_compile main.py
```

再用一个不存在的视频路径请求 `runninghub/2047787809091620866`，预期应在本地上传阶段返回“无法上传视频输入”，而不是再返回 RunningHub 的 `Invalid URL`。这说明代码已经进入高清模型专用分流。

## RunningHub 图片编辑工作流返回 Custom validation failed for node

### 现象

智能画布里选择 RunningHub 的 `NanoBanana-2-图片编辑` 或类似多图编辑工作流后，任务很快失败：

```text
RunningHub
NanoBanana-2-图片编辑
Custom validation failed for node
```

### 根因

这类 RunningHub 配置不是“随便给一堆参考图 + 一个提示词”的通用图片编辑接口，而是有固定图片槽位的工作流。

以 `NanoBanana-2-图片编辑` 为例，工作流 `2058541134623891458` 有 5 个图片输入槽，其中 `图1` 和 `图4` 是必填。默认提示词语义也是按槽位写的：

```text
图1的模特戴上图2的眼镜/图3的上衣/图4的裤子/图5的背包
```

如果用户只写 `身穿铠甲`，但一次带入很多参考图，前端会按顺序把图片塞进 `图1` 到 `图5`，多余图片不会进入工作流。RunningHub 内部节点发现必填槽位、图片语义或节点输入不符合工作流校验时，就会返回笼统的 `Custom validation failed for node`。

### 已修复

修复位置：`static/js/smart-canvas.js`

已给 RunningHub 工作流失败补充输入摘要。以后再出现这类错误，错误信息会同时带上：

- 当前工作流名称和 workflowId。
- `Image 1`、`Image 2`、`Image 3` 等槽位实际对应的图片。
- 哪些槽位是 required。
- 有多少多余图片被忽略。

这样可以直接判断是不是图片顺序、必填槽位或工作流语义用错，而不用只看 RunningHub 的模糊错误。

同时更新了 `static/smart-canvas.html` 里的脚本版本号，避免浏览器继续加载旧缓存。

### 防复发规则

以后接入或使用 RunningHub 工作流时，先看工作流字段配置：

1. 检查 `static/runninghub/api_providers.json` 中该 workflow 的 `fields`。
2. 找出所有 `fieldType=IMAGE` 的字段。
3. 按 `imageOrder` 确认 `图1/图2/图3...` 的含义。
4. 特别确认 `required=true` 的必填槽位。
5. 用户提示词必须和槽位语义匹配；不要把多图换装工作流当作单图编辑模型使用。
6. 一次输入的参考图超过工作流图片槽位时，只会使用前 N 张，多余图片不会进入 RunningHub。

### 处理建议

如果只是想把一张图里的主体改成“身穿铠甲”，优先使用单图编辑模型或只传目标图 + 明确的铠甲参考图。不要使用 `NanoBanana-2-图片编辑` 这种默认要求 `图1/图2/图3/图4/图5` 槽位语义的多图换装工作流，除非已经明确每张图对应哪个槽位。

### 2026-06-14 追加结论

后续验证发现，即使把 `NanoBanana-2-图片编辑` 的 5 个图片槽位全部填满，RunningHub 仍然返回：

```text
Custom validation failed for node
```

该 workflow 的主节点是 `RH_Nano_Banana2_Gemini31Flash`，不是普通稳定 OpenAPI 模型。失败已不再是前端漏传图片，而是 RunningHub 侧自定义节点校验未通过。

处理结果：

- 已将 workflow `2058541134623891458` 默认设为 `enabled=false`、`hidden=true`，避免继续出现在智能画布 RunningHub 工作流列表里。
- 同类图片编辑请优先使用 `GPT-Image-2-图片编辑` workflow，或使用普通 API 图片编辑模型。
- 不要再把这个 NanoBanana workflow 当作可用生产入口；除非后续从 RunningHub 重新拉取到可验证成功的新 workflow JSON，再恢复启用。

### 现象


### 根因

云雾 `gpt-image-2-all` 的图生图接口不是 OpenAI 标准 `/v1/images/edits`，而是：


Codex CLI 这条任务的日志显示图片系统拒绝了直接使用知名角色名，Codex 尝试改写为原创角色继续生成，但 5 分钟内没有产出图片。这里 5 分钟自动失败是预期保护，不是前端误判。

前端 `resumeSmartPendingNode()` 在恢复 pending 任务时，对普通失败仍会重新 `throw`，导致浏览器出现 `Uncaught (in promise)`，虽然后端状态已经正确失败。

3. 云雾 image2 的本地参考图按顺序处理：
   - 已经是公网 `http/https` 图片 URL：直接传给云雾。
   - 配置了 `PUBLIC_MEDIA_BASE_URL` 或 `PUBLIC_BASE_URL`：转换成本地媒体的公网 URL。
